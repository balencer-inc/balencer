# 知識システムの役割定義 — 5道具とデータの流れ（正本）

> バレンサーの知識・オペレーション基盤の唯一の役割定義。GitHub(git)・Obsidian・Notion・Google Drive・claude.ai/Claude Code の**どれが何を担い、情報がどう流れるか**をここで決める。
> 2026-07-15 制定（fableとの全体設計を反映）。以前の記述（`secretary.md` の1行・[[project-balencer-info-architecture]] の三層構造）はこれに統合。

## 0. 一行原則

**正本は git `docs/` 1箱。他の道具は全部「正本を見る/共有する/届ける」ためのインターフェース。マスターを増やさない。**

阿部さんの当初案「第二の脳を Obsidian に一元化する（Notion内容を全部 Obsidian へ移す）」は**採用しない**。理由は §2。

---

## 1. なぜ Obsidian マスターにしないのか（判断の記録）

阿部さんが Obsidian に求めたもの＝「ローカルに md がある・リンクでノードとして繋がる・AI が速く正確に読める」。**これは git `docs/` がすでに全部満たしている。** しかも git は claude.ai からも（クラウド面＝GitHub 経由で）読め、履歴もチーム共有もある。

一方 Obsidian をマスターにすると破綻する:
- **claude.ai はクラウドで動くのでローカル Obsidian を永久に読めない。** マスターを Obsidian に置いた瞬間、高品質対話の場から正本が見えなくなる。届けるには別クラウドへの同期＝「2つ目の編集できる真実」が必ず生まれる（＝二重化）。
- git は「ローカル作業コピー＋GitHub クラウドミラー」の同一コミットの2面。push は二重化ではなく鏡写し。**git マスターなら追加コピーはゼロ、Obsidian マスターは＋1。**

→ **答え：正本の所在（git）と、それを見る道具（Obsidian）を分離する。** Obsidian はただの md ビューア/エディタなので、`~/Projects/balencer/docs` を第2 vault として「開く」だけで、グラフ・リンク・ノード体験を正本の上でそのまま得られる。引っ越し・同期・二重化ゼロ。

## 2. 5道具の役割

| 場所 | 役割（1行） | AI との関係 |
|---|---|---|
| **git `balencer/docs/`** | **唯一のマスター**。会社定義・顧客・提案・決定・ナレッジ、AI が読むテキスト知識のすべて | Claude Code＝直読み直書き。claude.ai＝Project 経由（§4） |
| **Obsidian vault A**（`~/Documents/my-notes`） | 私的第二の脳（思想・メンバー非共有の数字）。**会社知識は置かない**。[[project-personal-obsidian-vault]] | Claude Code のみ。会社リポに絶対混ぜない |
| **Obsidian vault B**（`~/Projects/balencer/docs` を開く） | 正本の**閲覧・思考 UI**。グラフ・リンク・手書きメモ。中身は git の md そのもの | ツールであってストレージではない |
| **Notion** | **メンバーとの接触面だけ**（タスク・進捗・議事録・受信箱）。知識になったら git へ「昇格」しリンクを残す。保管庫にしない | AI は URL 指定の1ページのみ。DB 全 query・ツリー総なめ禁止 |
| **Google Drive** | **バイナリ専用**（Excel・PDF・動画・納品物・メンバー編集中ファイル） | git 側にポインタ md（リンク＋1行説明）を置き、AI は原則ポインタを読む。実体 DL は明示指示時のみ |
| **claude.ai / デスクトップ** | 高品質な**壁打ち・対話の場**（賢いモデルを使う） | 入力は Claude Project のナレッジ＋brief 手貼り（§4） |
| **Claude Code** | **構築・ファイル化・昇格のエンジン**。書き込みは全部ここを通す | 読み順: CLAUDE.md（自動）→ 対象フォルダ README → 必要ファイルのみパス指定 |

**正本の競合解消（明文ルール）**: 会社・顧客・仕事の正本＝git `docs/`（例外なし）。Obsidian vault A の正本性は「阿部さん個人の思想・私的数字」のみ。会社文書が vault A に紛れたら、それは下書きであって正本ではない。

## 3. データの流れ（顧客初回アポ準備＝オイシーズ型）

1. **素材集め＆ブリーフ生成**（Claude Code・安モデル）: `docs/clients/<slug>/` を確保（無ければ README 対応表に1行足して作成）→ Web 調査・NotebookLM 出力・Notion 該当ページ**だけ**URL 指定で取得 → `00_brief.md` を生成（**上限2,000字**、構成固定: ①相手は何者か ②当社との接点・経緯 ③課題仮説3つ ④当日の論点候補 ⑤参照リンク集）→ commit & push。
2. **高品質対話**（claude.ai・賢いモデル）: 当該 Project の新規チャット → `00_brief.md` を手貼り → 「ナレッジの CLAUDE.md／company/ と貼った brief だけで壁打ちして」。**リポ全読み・Notion 全読みは禁止。名指しだけ。**
3. **結論のファイル化**（Claude Code・安〜中モデル）: 対話の結論を貼る → `00_brief.md` に「## 壁打ち結論 YYYY-MM-DD」で追記、提案骨子は `01_提案/` に構築 → 決定・学びを `docs/_knowledge/<担当>.md` へ → commit & push → main。
4. **メンバー共有**（必要時のみ）: Notion に要約＋git への参照リンク。全文複製はしない。

## 4. claude.ai へ正本を届ける（本命：Claude Project の読み取り専用プロジェクション）

**GitHub 連携は不採用**（リポが重い＝`.git` 1.4GB／`apps/` 1.8GB／`node_modules/` 等で、AI が読みたくもないアプリ・ビルド・巨大履歴ごと掴もうとして詰まる）。**Notion/Drive 代替も不採用**（二重化を生む）。

**本命（※2026-07-15 時点で未検証。実地テストして効いたら CLAUDE.md §6 を更新する。未検証のまま §6 を上書きしない）**: claude.ai に Project を1つ作り、ナレッジに**安定コアだけ**を読み取り専用でアップロードする ── `CLAUDE.md` ＋ `docs/company/identity.md・services.md・results.md`（合計数十KB、滅多に変わらない＝腐らない）。

なお 2026-07-14 の検証で「claude.ai＋GitHub 追加」は技術的には繋がると確定済み（CLAUDE.md §6）。ただしリポが重く（§4冒頭）実用上詰まるため、本方式へ切り替えたい。GitHub 追加は Plan B として残す。

- これは**二重化ではない**。条件は3つ: (1)読み取り専用（誰もそこを編集して正本扱いしない） (2)向きは git→Project の一方通行、逆流させない (3)載せるのは滅多に変わらない安定コアのみ。→「正本から焼き直した使い捨てのプロジェクション（ビルド成果物）」。Notion が危険なのは*人が中で編集して勝手に育ち正本と食い違う*から。Project ナレッジは育たない。
- **顧客ごとの動的情報（brief）は Project に載せない。** その場で手貼り（2,000字だから一瞬）。
- **更新は一方通行・低頻度**: company/ を書き換えた週だけ、Claude Code が4ファイルを1バンドル md に書き出し → 阿部さんが Project にドラッグで差し替え（金曜棚卸しに1行）。

**避けられない事実**: claude.ai はクラウドなのでローカル Obsidian を永久に読めない。橋を架けるのではなく、対話に必要な軽い中核を正本から焼いて Project に置く ── これが制約を破らない唯一の筋。

（将来どうしても GitHub 連携を使いたければ、`apps/`・`out/`・LP 資産を別リポや Drive に出し、`node_modules` を git から外して軽くする Plan B がある。履歴圧縮は手間なので Project 方式で回してから。）

## 5. 精度の根本原因＝「正本が薄い」

AI は答える前に材料を集める。CLAUDE.md は「正本は `docs/company/` にある」と指さすが、そこがほぼ空だと**指さした先に物がない**。AI は答えを出すため、次に見つかる"それっぽい物"（旧 Notion の禁止表現・古い Drive デッキ）を拾って穴を埋める。**AI に「古い/正」の区別はつかないので、正本が空なら現存する旧資料が勝つ。** 掴んだ誤前提が提案・分析・コピーへ伝染する。

オイシーズで Claude Code の精度が悪かったのはこれ。逆に claude.ai で納得感が高かったのは、その場で阿部さんが口で正しい前提を供給していたから（対話では阿部さん自身が正本だった）。→ **教訓：対話をファイルに落とせば、次から毎回説明しなくても全セッションが良い前提を引き継ぐ。**

→ 対策は「別の道具へ引っ越す」ではなく **`docs/company/` を厚くすること**（§7 の宿題）。

## 6. 読ませ方・置き場・モデルの規律

**トークン節約（読ませ方）**
- Notion は「URL 指定した1ページ」以外読まない。検索総なめ禁止。
- git はパス指定 Read。フォルダ丸読みは README まで。
- 顧客 README は「5行サマリ＋リンク集」形式に統一（最初の1読で全体像）。
- Drive は実体を読まずポインタ md で済ます。実体 DL は明示指示時のみ。

**資料の置き場（判定は1つ）**: 「**次の AI セッションに読ませたいか？**」
- 読ませたい md/HTML/コード → **git**（AI の再利用資産になる）
- 見せるだけの Excel・PDF 最終納品物・メンバー手編集ファイル → **Drive**（＋git にポインタ md）

**モデルの使い分け**: 原則「**書き込み先が正本（`docs/company/`・CLAUDE.md）なら賢いモデル、それ以外は安いモデル**」。
- 賢いモデル: claude.ai の壁打ち、正本ドキュメント初版、CLAUDE.md 改訂
- 中間: 提案書の本文化、調査要約
- 安モデル: brief 下ごしらえ、Notion→md 変換、整形・リンク張り、commit 作業、検品チェックリスト実行
