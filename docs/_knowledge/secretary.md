## 2026-07-14 claude.ai と Claude Code の役割分担・顧客相談の入口

**背景**: 阿部さんは相談・壁打ちを claude.ai の方が精度高く感じる一方、実作業は Claude Code。claude.ai に「顧客の過去を分かった状態」で相談を始めたいが、毎回md を渡すのは手間。DriveやNotionに顧客情報を二重保存する案も出た。

**決定（道具の役割を固定）**:
- **GitHub(git) = 唯一の正本**（会社・顧客・決定の記憶）
- **Claude Code = 手**（正本を読み書きし作る・コミット）
- **claude.ai = 頭**（壁打ち・相談。正本を"読むだけ"）
- **Notion = チーム運用面**（タスク・進捗）／**Drive = gitに向かない物**（Excel・大ファイル・納品物）／**Obsidian = 私的第二の脳**
- → **却下: 顧客情報をDrive/Notionに二重保存**。正本が2つになり「どっちが最新」問題を生む。二重管理は作らない。

**やり方（claude.ai 側・チームプラン=全員有料で利用可）**:
- 顧客ごとに claude.ai で**プロジェクトを1つ**作り、**GitHubコネクタで `balencer` リポジトリを接続**
- プロジェクトのカスタム指示に「回答前に `CLAUDE.md` と `docs/clients/<スラッグ>/README.md` を読む。数字/事実はリポジトリ記載を正とする」を貼る
- **顧客READMEをそのまま入口にする**（専用の `_context.md` は作らない=北麓READMEと二重になるため却下）
- 相談で決まったことは Claude Code 側でファイルに落としてコミット → 次回 claude.ai も最新を読む

**スラッグ揺らぎ対策**: `docs/clients/README.md` に「顧客スラッグ対応表（唯一の正）」を新設。claude.ai相談時も新規フォルダ作成時も、まずこの表でスラッグを引く。新規顧客は表に1行足してからフォルダを作る。

## 2026-07-14（追記）テストで判明・方針転換とgit迷子問題

**テスト結果**: クラビスのプロジェクトをclaude.aiで作り「設立年月と3期売上」を質問 → **main にある CLAUDE.md すら読めず**、claude.aiは正直に「GitHubを読むツールが無い」と報告（＝ハルシネーションせず満点の挙動）。GitHub接続は機能していなかった。

**判明した2つの根本問題**:
1. **claude.aiのGitHub接続が効かない**。かつ阿部さんは相談を **Claude Desktop（Macアプリ）** で行っている。
2. **git迷子（本丸）**: origin/main が 6/14 で止まり、以降の顧客仕事が **47コミット × 6ブランチ**に分裂して main 未マージ（ai-training-proposal 55・tsugi-ma-hearing 11 等）。「正本はgit」なのに main が実態を映していなかった。CLAUDE.md §5に「新ブランチで作業」はあったが「終わったら main に戻す」が抜けていたのが原因。

**方針転換（重要）**:
- **却下: claude.aiのGitHubコネクタ**（同期ラグ・main読み・接続が不安定）。
- **採用: Claude Desktop のファイルシステム連携で `~/Projects/balencer/` を直読み**。ローカル正本をそのまま読むのでラグゼロ・常に最新・二重保存なし。※要: Desktopのフォルダ許可を Desktop だけ→ `~/Projects/balencer` に拡張。
- 前提として **main を最新に保つ運用**が必須 → CLAUDE.md §5 に「作業後 main へマージして戻す／作業前にブランチ確認」を追記、§6 に相談時の正本読ませ方を新設。

**実施**: 現作業ブランチ（zealous, 47コミット）を origin/main へ早送り集約し main を7/14最新化。残り5ブランチ（ai-training-proposal 等）は生死の仕分けが別途必要（未実施・阿部さん判断待ち）。52ad5134（クラビス契約書v1）は obsidian-workflow に保全済み。
