# docs/clients/ — 顧客案件の作業場（1顧客1フォルダ）

> 秘書受付で「顧客の仕事」と判定されたら、**必ずこの下の顧客フォルダで作業する**。
> フォルダが無ければ作ってから始める。作業が終わったら記録を残す。この流れを崩さない。

## 受付からの流れ

1. 用件が顧客の仕事 → `docs/clients/<顧客スラッグ>/` を確認
2. **あればそこで作業。なければフォルダを作って作業**（ルート直下や別の場所に散らさない）
3. 終わったら: 成果物をフォルダ内に保存 → 顧客フォルダの `README.md` に一行追記（何をしたか）→ `docs/_knowledge/<担当>.md` に知見を記録 → コミット＆プッシュ

## 命名規則

- スラッグはローマ字ケバブケース: `adachi-onken` `hokuroku` `nailit` `mutsubi` `1stplace`
- 同じ顧客の別案件でもフォルダは1つ（案件はサブフォルダで分ける）

## フォルダ内の型（必要になったものだけ作る）

```
<顧客スラッグ>/
  README.md      ← 顧客の基本情報と作業履歴（一行ログ）。「5行サマリ＋リンク集」形式
  00_brief.md    ← claude.ai 壁打ち用の入口ブリーフ（2,000字上限）。壁打ち結論もここに追記
  01_提案/
  02_契約・見積/
  03_議事録/
  04_成果物/
  05_素材/
```

※Google Driveのクライアントフォルダ標準構造（drive-cleanup-plan-2026-07.md STEP 4）と同じ番号体系。

## README と 00_brief.md の型（入口を軽く保つ）

**README.md** はAIが最初の1読で全体像を掴めるよう「5行サマリ＋リンク集」に保つ:
```
# <正式名>（<スラッグ>）
- 何者か: （業種・規模・キーパーソン）
- 当社との関係: （いつから・何を・現在のフェーズ）
- 直近の状況: （最新の論点・宿題）
- 数字/機微: （あれば置き場のリンクだけ。本文に生数字を貼らない）
## リンク集
- 提案 → 01_提案/ ／ 見積 → 02_契約・見積/ ／ 議事録 → 03_議事録/ …
- Drive実体 → （ポインタ。リンク＋1行）
```

**00_brief.md**（claude.ai の Project チャットに手貼りする入口ブリーフ・**上限2,000字**）:
```
# <顧客> 壁打ちブリーフ（YYYY-MM-DD）
① 相手は何者か（5行）
② 当社との接点・経緯
③ 相手の課題仮説（3つ）
④ 当日の論点候補
⑤ 参照リンク集（Notion/Driveポインタ）

## 壁打ち結論 YYYY-MM-DD
（claude.ai での対話結果をここに追記 → Claude Code が提案骨子へ展開）
```

> 詳しい流れ（素材集め→brief→claude.ai壁打ち→ファイル化）は `docs/_knowledge/knowledge-system.md` §3。

## 既存の置き場との関係

- 過去の提案書 `docs/proposals/` ・見積 `docs/estimates/` は**そのまま**（移動しない）。新規案件からこの構造を使う
- Vercel公開などの事情で成果物が別フォルダに置かれる場合は、顧客READMEからリンクを張る
- 顧客の機微情報（決算書・個人情報）はコミット前に要判断。迷ったら阿部さんに確認
