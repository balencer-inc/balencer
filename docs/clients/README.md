# docs/clients — 顧客ごとの案件フォルダ

> セッションをまたいでも **顧客情報が先祖返りしない** ための置き場。
> 1顧客＝1フォルダ。各フォルダの `README.md` が「その顧客の現在地」＝**再開の起点**。
> 新しいセッションは、まずここを見れば「誰の・何の話で・どこまで進んだか」が分かる状態にする。

---

## セッションの型（秘書が必ず通す）

**開始**
1. 「**どの案件の話か**」を最初に確認する：
   - 社外クライアント（例: 北麓 / nailit …）
   - 自社（**バレンサー** ／ **ツギアンドパートナーズ**）
   - それ以外（新規・社内雑務など）
2. 対応するフォルダを開く。あれば既存の `README.md` を読んで現在地を把握してから着手。無ければ `_TEMPLATE/` を複製して作る。
3. 用件に合う担当（`Skill`）へ渡す。

**終了**
4. 成果物をその顧客フォルダに保存。
5. その顧客の `README.md` を**最新の現在地に更新**（下の項目を埋める）。
6. **コミット＆プッシュ**（会話だけで終わらせない）。

---

## 置き場のマップ

| バケット | 置き場所 |
|---|---|
| 社外クライアント | `docs/clients/<client>/`（このフォルダ） |
| 自社：バレンサー | 定義・戦略は `docs/company/` ／ `docs/handoff/`、LP・制作は `docs/<プロダクト>-lp/` 等 |
| 自社：ツギアンドパートナーズ | `docs/clients/tsugi-and-partners/`（無ければ作る。※事業承継・TSUGI関連） |
| 担当別の横断ナレッジ | `docs/_knowledge/<担当>.md`（顧客横断の知見・型・好み） |

> 顧客フォルダ名は**ローマ字小文字**（読み準拠）。例：北麓＝`hokuroku`。
> 顧客固有の「現在地・経緯」は各フォルダの `README.md`、担当横断の「型・学び」は `docs/_knowledge/` に分けて書く。
