# トラベルダイン システム現状レポート（保守引き継ぎ検討用）

- 作成日: 2026-07-22
- 作成: 株式会社バレンサー
- 対象システム: トラベルダイン（訪日インバウンド向けの飲食店マッチングサイト）
- 対象リポジトリ: `partner-hare-inbound-be`（サーバー側）／`partner-hare-inbound-fe`（画面側）
- 目的: 保守引き継ぎを検討するにあたり、システムの構成・規模・現在の開発/保守の活動量・保守に必要な工数を事実として把握する
- 確認方法: 共有いただいたアクセス権で両リポジトリを取得し、ソースコード・設定・変更履歴を確認した

> 本レポートは、バレンサーの関わり方や契約・費用の話は含みません（別途ご相談します）。あくまでシステムの事実の把握を目的としています。

## 1. サマリー

- トラベルダインは、**サーバー側（PHP／Laravel）と画面側（Vue／Nuxt）の2つのシステムで構成**される、本番稼働中のWebサービスです。
- 2023年7月から開発が始まり、**現在も継続的に改修が行われています**（直近も日次に近い頻度で更新あり）。
- 変更履歴を見ると、機能は本格的で（予約・多言語対応・LINEログイン・リアルタイム通信・帳票出力など）、**専門の開発チームが継続的に関わることを前提とした規模・作り**です。
- 現在の改修は、主に**課題管理システム（pms.haposoft.com）のチケット単位**で進められています。
- ソースコード内に、パスワードやAPIキーなどの秘密情報が直接書き込まれている箇所は確認されませんでした（適切に管理されています）。
- **バレンサーで引き継ぐことは技術的に可能**です。ただし必要な体制・費用は「保守の範囲」によって変わるため、第7章の確認事項をふまえて範囲を確定させたうえでの引き受けになります（結論は第8章）。

## 2. システム構成（何でできているか）

トラベルダインは2つのシステムに分かれています。

### サーバー側 `partner-hare-inbound-be`
利用者から見えない、データの処理・保存・連携を担う部分です。

| 項目 | 内容 |
|---|---|
| 開発言語・基盤 | PHP 8.1 ／ Laravel 9（広く使われている標準的なWeb開発の枠組み） |
| 規模 | プログラムファイル 約636個 ／ データベースの構造定義 290件 |
| 主な機能 | 会員認証、LINEログイン、リアルタイム通信（メッセージ等）、PDF帳票の出力、AWS（クラウド）へのファイル保存、Googleスプレッドシート連携、多言語対応 |
| 品質・運用の仕組み | 自動テスト、コード品質の自動チェック（SonarQube等）、データベースの負荷分散（読み取り専用の複製）、自動デプロイ（サーバーへの反映）、Dockerによる開発環境 |
| 説明資料 | 環境構築手順（README）・ローカル動作確認ガイド・負荷分散の実装メモが用意されている |

### 画面側 `partner-hare-inbound-fe`
利用者が実際に見て操作する画面の部分です。

| 項目 | 内容 |
|---|---|
| 開発言語・基盤 | Vue 3 ／ Nuxt 3（TypeScript）（画面づくりの標準的な枠組み） |
| 規模 | 画面・部品ファイル 約136個 |
| 主な機能 | 管理画面のUI部品（Element Plus）、カレンダー表示、状態管理、多言語対応、サーバー側とのリアルタイム連携 |

いずれも、現在広く使われている標準的な技術で作られており、特殊で扱いにくい技術は使われていません。

## 4. 保守に必要な工数の見立て

「保守」といっても、どこまでを対象にするかで必要な工数が大きく変わります。変更履歴から読み取れる事実をもとに、2つのケースで目安を示します（あくまで目安であり、正確な数値は下記「前提・注意」を要確認）。

### ケースA：現状のペースを維持する（改修・小さな機能追加を継続）
現在の活動量をそのまま引き継ぐ場合の目安です。

- 実績ベースの活動量：サーバー側で月30〜50件、画面側で月7〜9件の改修が反映されている
- これは**開発者2〜3名相当が継続的に関わっている**水準です
- 引き継ぐ場合、同等の体制（PHP/LaravelとVue/Nuxtの両方を扱える人員）が必要になります

### ケースB：最小限の保守にとどめる（障害対応・セキュリティ更新・軽微な修正のみ、新機能は行わない）
サービスを止めずに維持することだけを対象にする場合の目安です。

- 新規開発を含まないため、工数はケースAより大幅に小さくなります
- ただし、**この大きさのシステムを安全に維持するには、コード全体を理解した担当が必要**です。障害はいつ起きるか読めないため、対応できる体制を用意しておくこと自体にコストがかかります
- 現在このシステムは「最小限の保守」モードでは運用されておらず、継続的に改修が入っている状態です

### 工数を正確に出すために必要な追加情報
現時点の変更履歴からは「改修＝新機能」と「保守＝不具合修正・維持」の内訳までは正確に切り分けられません。正確な工数を出すには、以下の情報が有効です。

- 課題管理システム（pms.haposoft.com）の**チケットの内訳**（新機能／不具合／問い合わせ対応の割合と件数）
- 過去1年の**障害・緊急対応の発生頻度**
- 現在の運用体制（何名が、月あたりどれくらいの時間を充てているか）

## 5. 引き継ぎにあたって事実として必要になること

技術面で、引き継ぎ時に必要になる作業を事実として列挙します（進め方の相談は別途）。

- **アクセス権の正式な移管**：現状は担当者アカウントの共有。引き継ぐ場合は、引き継ぎ先を正式なメンバーとして登録する形が安全です
- **サーバー・ドメイン・外部サービスの情報**：本番サーバー、データベース、AWS、LINE連携、Googleスプレッドシート連携などの管理情報の引き継ぎ
- **開発環境の再現**：手順書（README）は用意されているため、それに沿って手元で動かせる状態を作る作業
- **課題管理システムの引き継ぎ**：現在の運用が pms.haposoft.com に紐づいているため、今後どこで課題管理するかの整理

## 6. 引き継ぎにあたって確認したい事項

引き継ぎの範囲と体制を正しく見積もるため、以下を確認させてください。ここが定まると、必要な体制・進め方が具体化できます。

1. **現在の保守の範囲と費用の中身** — 現在の月次保守費用は、「システムを止めない維持（障害対応・軽微な修正）」が中心か、それとも「新機能の開発」も含んでいるか。第3章のとおり直近も継続的に改修が入っているため、この切り分けが体制・費用を大きく左右します。
2. **課題（チケット）の内訳** — pms.haposoft.com の直近1年について、新機能／不具合修正／問い合わせ対応のおおよその割合と件数。
3. **障害・緊急対応の頻度** — 過去1年で、緊急対応が必要になった回数の目安。
4. **今後の開発体制** — 現在の開発チーム（ハポソフト）は今後も継続するのか、引き継ぎに伴い縮小・終了するのか。
5. **アクセス権・インフラの移管可否** — GitHub・本番サーバー・データベース・外部連携（AWS／LINE／Googleスプレッドシート）の管理情報を引き継げるか。

## 7. 結論（引き継ぎの可否）

- 技術的には、標準的な構成のシステムのため、**バレンサー（＋パートナーのエンジニア）で保守を引き継ぐことは可能**です。日本語での密な連携ができ、将来の社内DXも同じ体制で見据えられる点が利点です。
- そのうえで、引き受けの体制と費用は「保守の範囲」によって変わります。**第6章の確認が取れ、範囲が「維持を中心とした保守」で合意できれば、現行の枠組みの中で引き継ぐことが可能**です。
- 新機能の開発まで含めて継続する場合は、必要な人員が増えるため、体制・費用を改めて設計します。
- いずれの場合も、まずは第6章の確認をお願いできればと思います。そのうえで、具体的な進め方・費用を別途ご提案します。
