# 自社ファネル設計図｜診断 → 相談 の常設導線

> [受注方程式](juchu-hoteishiki.md) の②診断→③相談を、単発でなく**24時間動き続ける常設導線**として繋ぐ設計。
> 方針: ゼロから作らない。**補助金系で実証済みの「受け皿＋追客エンジン」を土台に、不足だけ足す**。
> 関連: server-scripts/（PHP実体）／[project_hojokin_hub_lp]／apps/outreach（追客の別実装）

> **【2026-07-15 訂正】** 本ドキュメント初版は git内の古いLPだけを見て「診断が未整備・予約検知が弱い」と誤認していた。実態は次の通り修正: **①診断は完成済み・稼働中**＝「経営者の右腕診断」(balencer.jp/shindan/・4領域20問→4タイプ＋スコア→詳細レポートをメール)。Clavisの「立て直し型53/100」はこれ。**②予約もGoogleカレンダーで稼働中**（診断後に予約ルームが出てオンライン相談まで繋がる）。したがって「診断→予約→オンライン相談」は既にlive。残る本当のギャップは下記§4で再定義。全体像は [自社セールスパイプライン全体像](sales-pipeline-overview.md) を参照。

---

## 1. 目的（何を繋ぐか）

Clavisでは「紹介→診断→相談→契約」が回った。ただし診断も相談も**その都度・手動**だった。これを常設化する。狙いは2つ:
- **診断で終わらせない** ── 診断した人を必ずリードとして捕まえ、相談予約まで自動で運ぶ
- **取りこぼさない** ── 予約しなかった人を自動で追いかけ、予約が入ったら止める。状況はダッシュボードで見える

---

## 2. 目標フロー

```
[入口]                [診断]              [受け皿/リード]        [相談]           [追客/可視化]
紹介・SNS・ブログ  →  無料AI組織診断  →  結果メール＋Notion  →  Zoom相談予約  →  +3/7/14日 追客
                     (数分・スコア型)    DB登録＋Slack通知      (Calendar/Zoom)   予約で自動停止
                                                                              ＋ダッシュボード
```

## 4. 4つのギャップと繋ぎ方

### ギャップA｜診断が常設入口になっていない → 🔨作る＋🔗繋ぐ
- **やること**: 自社サイトに常設の「無料AI組織診断」を1本置く（Clavisで効いた"立て直し型/スコア型"を踏襲）。設問→スコア→タイプ判定→**結果表示の直前でメール取得**
- **繋ぎ**: 診断のsubmit先を新規 `server-scripts/shindan-lead.php`（hojokin-lead.phpを複製し、診断スコア/タイプをペイロードに追加）にする。結果メールに診断結果＋Zoom予約リンクを同梱
- **注意**: 診断結果は先に画面で見せてもよいが、**詳細レポート/相談は要メール**にして必ずリード化する

### ギャップB｜診断結果→リードDBにスコアが入らない → 🔗繋ぐ
- **やること**: Notion DBに「診断タイプ」「スコア」「弱点カテゴリ」列を足す。shindan-lead.phpから書き込む
- **効果**: 相談前に相手の課題が分かる＝町医者型相談の質が上がる（Clarisの勝ち筋の再現）

### ギャップC｜予約の検知ができず追客が止まらない → 🔨作る
- **問題**: 今は booking_url に飛ばすだけで、予約が入っても `booked=true` にする経路がない → 予約済みの人にも追客が飛ぶ恐れ
- **やること（いずれか）**:
  - (推奨) Zoom Scheduler/Google Calendar予約時のWebhook or メール通知を受け、該当リードのJSON/Notionを `booked=true` に更新する小さなendpoint
  - (簡易) 予約完了ページに「予約完了」フラグ用の軽いping（thanksページ→mark-booked.php）
- **同時に**: 予約が入ったら**Slack通知**（現状は社内メールのみ）。担当の対応漏れ防止

### ギャップD｜ファネルが見えない → 🔨作る
- **やること**: Notion DBのビュー（診断数／リード数／予約数／商談数）をダッシュボード化。まずはNotionのボード/集計ビューで十分。将来はapps/outreach側に統合も可
- **効果**: 「診断→予約」の歩留まりが見え、どこを直すか判断できる

## 5. 実装順（最小で回る所から）

1. **shindan-lead.php を作る**（hojokin-lead.php複製＋診断ペイロード対応）＝受け皿を先に用意
2. **常設「AI組織診断」LP/UIを1本作る**（診断→結果→メール取得→予約リンク）
3. **Notion DBに診断列追加＋cron追客を診断リードに接続**（hojokin-cron.php流用）
4. **予約検知＋Slack通知**（ギャップC）
5. **ダッシュボード**（Notionビュー）

→ 1〜3で「診断したら自動でリード化され追客される」最小ループが回る。4〜5は運用しながら足す。

## 6. 未確認・要判断（着手前に潰す）

- 診断の**設問とスコアロジック**の正典が必要（Clavisの「立て直し型53/100」の元ロジックを流用できるか要確認。既存LPのJSにあるはず → 発掘）
- 相談予約は**Zoom Scheduler / Googleカレンダー予約 / Calendly**のどれを常設にするか（予約検知のしやすさで選ぶ）
- Notion DBは**補助金と同じDBに相乗り**するか**自社集客用に新設**するか（混ざると集計が濁るので新設推奨）
- 診断の設置場所＝**balencer.jp本番(WP+Elementor)にFTPで静的設置**（[reference_balencer_live_site]の.htaccess素通り方式）
