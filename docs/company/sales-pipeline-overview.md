# 自社セールスパイプライン全体像

> バレンサーが「知らない人と出会って→契約→コンサル開始」までの全体像を1枚で持つための地図。
> 頭に入れる用の俯瞰図。個別の実装設計は各リンク先へ。**他社の営業パイプライン設計にも流用できる汎用の型**として整理する。
> 関連: [受注方程式](juchu-hoteishiki.md)／[診断→相談の設計](funnel-shindan-soudan-design.md)／[事例](case-studies/case-rm-support.md)

---

## 1. 全体の流れ（5ステージ＋横断するナーチャリング層）

```
① 流入            ② 診断              ③ 予約            ④ クロージング       ⑤ コンサル開始
知ってもらう    →  現在地を見せる   →  会う約束を取る  →  オンラインで決める → 伴走スタート
                                                                            
紹介 / SNS /       経営者の右腕診断     Googleカレンダー    初回オンライン →     契約書 → 6ヶ月伴走
ブログ / 名刺      (4領域20問・5分)     予約ルーム          2回目で契約         毎月「使えるもの」1納品
                   → スコア＋タイプ     → オンライン相談    (自社実例を見せる)
                   → レポートをメール

────────────────────────────────────────────────────────────────────────
        ⟳ ナーチャリング層（横断）：メルマガ / ステップメール / 眠り名刺の掘り起こし
          「今すぐ客」でない人を、関係を切らさず温め続けて①〜③へ戻す
```

---

## 2. ステージ別ステータス（いま何が動いているか）

| ステージ | 中身 | 現状 |
|---|---|---|
| ①流入 | 母数をつくる。紹介・SNS・ブログ・名刺・著書 | 🟡 **紹介と著書は強い。ブログ/SEOはほぼ未着手＝母数の入口が細い**。Clavisも入口は紹介 |
| ②診断 | 経営者の右腕診断（balencer.jp/shindan/）。4領域20問→4タイプ＋スコア→詳細レポートをメール | 🟢 **稼働中**。Clavisの53/100はこれ。勝ち筋の核 |
| ③予約 | Googleカレンダー予約ルーム→オンライン相談 | 🟢 **稼働中**（診断後に予約→オンラインまで繋がる） |
| ④クロージング | 初回オンライン→2回目で契約。診断結果を入口に、自社実例で信頼、相手の特殊性に寄り添う | 🟢 型は実証済（[受注方程式](juchu-hoteishiki.md)）。属人的＝阿部さんの対話力に依存 |
| ⑤コンサル開始 | 契約書→6ヶ月伴走→毎月成果物 | 🟢 型あり（ENTRY/MAIN/PREMIUM・契約書テンプレ） |
| ⟳ナーチャリング | メルマガ・ステップメール・眠り名刺 | 🔴 **ほぼ未整備＝最大の伸びしろ**。名刺交換したが一度も配信していない眠り資産が多い |

> 結論: **「診断→予約→相談→契約」の背骨はもう立っている**。伸びしろは両端＝**①入口の量（ブログ/SEO）**と**⟳ナーチャリング（メルマガ・眠り名刺）**。ここが繋がると背骨に人が流れ続ける。

## 3. ナーチャリング層（メルマガ・眠り名刺）── 次の主戦場

### なぜここか
背骨（診断→契約）は完成しているのに、そこへ**継続的に人を送り込む仕組みがない**。特に:
- **眠り名刺**: 交流会等で名刺交換した相手に一度も配信していない。今すぐ客でなくても、温め続ければ①〜③に戻ってくる母集団
- **診断リード**: 診断はしたが予約しなかった人を、メルマガで関係維持 → 再来訪

### 設計の骨子（要検討）
1. **リストの源泉**: ①名刺（交流会・商談）②診断リード（予約しなかった人）③問い合わせ・DL
2. **配信基盤**: メルマガ配信サービスを使うか、自前(apps/outreach = Resend + Supabase)で組むか → §5で判断
3. **配信内容**: 売り込みでなく「使える一次情報」（受注方程式の"押し売らない"を踏襲）。SNS運用の型と共通の弾を流用
4. **同意・法務**: 名刺交換だけを根拠にした一斉配信は特定電子メール法のオプトイン論点あり → 初回に許諾/オプトアウト導線を必ず設計

---

## 4. 他社への流用（汎用パイプラインの型）

このマップの5ステージ＋ナーチャリング層は、バレンサー固有情報を抜けば**そのまま他社の営業パイプライン設計テンプレ**になる。他社案件では各ステージに「その会社の①〜⑤は何か」を埋める。診断＝フック、予約＝会う約束、ナーチャリング＝温め、の役割は業種を問わない。

## 5. 着手前に潰す宿題（未確認・要判断）

- ❓ **診断リードの行き先**: balencer.jp/shindan/ で取得したメアド/回答は、いまどこに溜まっているか（WPフォーム？Notion？スプレッドシート？）。ナーチャリングに繋ぐ前に要確認
- ❓ **予約後の自動化**: Googleカレンダー予約が入ったとき、Slack通知や追客停止は自動か手動か
- ❓ **メルマガ基盤の選定**: 阿部さんが聞いた「ディセンド（要・正式名称確認）」を含め、配信サービスと自前実装(Resend)を比較して決める → **NotebookLMで調査予定**（[[feedback_research_use_notebooklm]]）
- ❓ **眠り名刺のデータ化**: 名刺は今どこに（Sansan/Eight/紙/エクセル）。配信リスト化の手間を見積もる
